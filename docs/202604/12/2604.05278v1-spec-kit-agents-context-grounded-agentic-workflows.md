## 摘要
使用 AI 编程智能体的规范驱动开发 (SDD) 提供了一种结构化的工作流，但智能体在大型且不断演进的代码库中往往处于“上下文盲区”，从而导致虚构 API 和架构违规。我们提出了 Spec Kit Agents，这是一个包含产品经理 (PM) 和开发人员角色的多智能体 SDD 流水线，它引入了阶段级的上下文增强钩子 (context-grounding hooks)。只读探测钩子将每个阶段（规范、计划、任务、实现）锚定在代码库证据中，而验证钩子则根据环境检查中间产物。我们在五个代码库中对涵盖 32 个功能的 128 次运行进行了评估。实验结果显示，上下文增强钩子在 1-5 分的 LLM 综合评分中将质量提升了 0.15 分（占总分的 3.0%；Wilcoxon 符号秩检验，p < 0.05），同时保持了 99.7-100% 的代码库级测试兼容性。我们进一步在 SWE-bench Lite 上评估了该框架，增强钩子使基准性能提升了 1.7%，达到了 58.2% 的 Pass@1。

## Abstract
Spec-driven development (SDD) with AI coding agents provides a structured workflow, but agents often remain "context blind" in large, evolving repositories, leading to hallucinated APIs and architectural violations. We present Spec Kit Agents, a multi-agent SDD pipeline (with PM and developer roles) that adds phase-level, context-grounding hooks. Read-only probing hooks ground each stage (Specify, Plan, Tasks, Implement) in repository evidence, while validation hooks check intermediate artifacts against the environment. We evaluate 128 runs covering 32 features across five repositories. Context-grounding hooks improve judged quality by +0.15 on a 1-5 composite LLM-as-judge score (+3.0 percent of the full score; Wilcoxon signed-rank, p < 0.05) while maintaining 99.7-100 percent repository-level test compatibility. We further evaluate the framework on SWE-bench Lite, where augmentation hooks improve baseline by 1.7 percent, achieving 58.2 percent Pass@1.

论文针对当前 AI 编码智能体（Coding Agents）在处理大型、动态演进的代码库时存在的**“上下文盲区”（Context Blindness）**问题。尽管现有的规范驱动开发（SDD）流程（如 GitHub 的 Spec Kit）通过引入中间产物（规范、计划、任务列表）来增强逻辑性，但智能体仍经常生成与实际代码库不兼容的 API、引用不存在的文件路径或违反架构规范。论文旨在通过一种**上下文增强（Context-Grounding）**的机制，使智能体在生成代码前，其思考过程能与代码库的真实状态强制对齐。

### 2. 方法论：核心思想与关键技术
论文提出了 **Spec Kit Agents**，一个多智能体（PM 角色 + 开发者角色）流水线，其核心是在 SDD 的四个阶段（Specify, Plan, Tasks, Implement）中嵌入了**上下文增强层**：

*   **多智能体协作**：PM 负责需求澄清，开发者负责生成中间产物（`SPEC.md`, `PLAN.md`, `TASKS.md`）并实施代码。
*   **发现钩子（Discovery Hooks）**：在每个阶段**开始前**运行。通过只读工具（如 grep, git history, globbing）探测代码库，收集 API 约定、依赖关系和模块信息，作为生成该阶段产物的上下文。
*   **验证钩子（Validation Hooks）**：在每个阶段**结束后**运行。检查中间产物的合法性（如计划中的文件路径是否存在、库是否已安装），并在最终实现阶段运行测试（pytest, linter）以确保无回归错误。
*   **权限控制**：采用最小权限原则，发现钩子仅限只读，验证钩子在必要时获得执行权限。
