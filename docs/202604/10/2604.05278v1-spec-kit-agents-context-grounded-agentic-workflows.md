## 摘要
使用 AI 编程智能体的规范驱动开发 (SDD) 提供了一种结构化的工作流，但在大型且不断演进的代码库中，智能体往往处于“上下文盲”状态，导致产生虚构的 API 和违反架构规范。我们提出了 Spec Kit Agents，这是一个多智能体 SDD 流水线（包含产品经理和开发人员角色），它增加了阶段级的上下文对齐钩子。只读探测钩子将每个阶段（规范、计划、任务、实现）锚定在代码库证据中，而验证钩子则根据环境检查中间产物。我们在五个代码库中对涵盖 32 个功能的 128 次运行进行了评估。实验结果表明，上下文对齐钩子在 1-5 分的 LLM 综合评分中将质量提升了 0.15 分（占总分的 3.0%；Wilcoxon 符号秩检验，p < 0.05），同时保持了 99.7-100% 的代码库级测试兼容性。我们进一步在 SWE-bench Lite 上评估了该框架，增强钩子将基准性能提升了 1.7%，达到了 58.2% 的 Pass@1。

## Abstract
Spec-driven development (SDD) with AI coding agents provides a structured workflow, but agents often remain "context blind" in large, evolving repositories, leading to hallucinated APIs and architectural violations. We present Spec Kit Agents, a multi-agent SDD pipeline (with PM and developer roles) that adds phase-level, context-grounding hooks. Read-only probing hooks ground each stage (Specify, Plan, Tasks, Implement) in repository evidence, while validation hooks check intermediate artifacts against the environment. We evaluate 128 runs covering 32 features across five repositories. Context-grounding hooks improve judged quality by +0.15 on a 1-5 composite LLM-as-judge score (+3.0 percent of the full score; Wilcoxon signed-rank, p < 0.05) while maintaining 99.7-100 percent repository-level test compatibility. We further evaluate the framework on SWE-bench Lite, where augmentation hooks improve baseline by 1.7 percent, achieving 58.2 percent Pass@1.

### 1. 核心问题与研究动机
*   **核心问题**：现有的 AI 编程智能体在处理大型、持续演进的代码库时，常面临“**上下文盲（Context Blindness）**”的问题。
*   **研究背景**：尽管规范驱动开发（SDD）通过生成中间产物（如需求规范、计划）来增强逻辑性，但智能体仍容易产生“幻觉”，例如引用不存在的 API、建议错误的路径或违反项目特有的架构规范。这些错误在后期实现阶段会产生连锁反应，导致开发失败或效率低下。

### 2. 方法论：Spec Kit Agents 框架
*   **核心思想**：在多智能体工作流中引入一个**上下文关联层（Context-Grounding Layer）**，通过在每个开发阶段前后插入“钩子（Hooks）”，强制智能体与代码库的实际状态进行同步。
*   **多智能体角色**：
    *   **产品经理 (PM)**：负责澄清需求和优先级。
    *   **开发人员 (Developer)**：负责生成中间产物（SPEC.md, PLAN.md, TASKS.md）并编写代码。
*   **关键技术细节（双钩子机制）**：
    *   **发现钩子 (Discovery Hooks)**：在每个阶段开始前运行，执行只读探测（如 grep、git history、文件搜索），为智能体提供当前代码库的真实证据。
    *   **验证钩子 (Validation Hooks)**：在每个阶段结束后运行，检查中间产物的合法性（如 PLAN.md 中的路径是否存在、依赖是否可用），并在代码实现后运行测试和静态检查（Linter）。
*   **工作流流程**：指定规范 (Specify) → 制定计划 (Plan) → 分解任务 (Tasks) → 执行实现 (Implement)。
