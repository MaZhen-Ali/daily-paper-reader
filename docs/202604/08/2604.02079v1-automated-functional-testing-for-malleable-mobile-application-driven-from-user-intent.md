## 摘要
软件可塑性允许应用程序在部署后仍能被轻松更改、配置和适配。尽管先前的工作已经探索了可配置系统、自适应推荐系统和可塑性图形用户界面（GUI），但这些方法通常针对特定软件定制，缺乏通用性。在这项工作中，我们设想了面向每个用户的可塑性移动应用，终端用户可以指定需求，并通过基于大语言模型（LLM）的代码生成自动实现。然而，实现这一愿景需要克服一个关键挑战，即设计自动化测试生成，以可靠地验证用户指定功能的存在性及其正确性。我们提出了 \tool，这是一个用户需求驱动的 GUI 测试生成框架，它增量式地导航 UI、触发所需功能，并构建 LLM 引导的测试断言（oracles）以验证正确性。我们构建了一个涵盖六款流行移动应用的基准测试，包含正确和错误的用户请求功能，证明了 \tool 能够有效验证每个用户的功能特性，并具有实际部署的可行性。我们的工作强调了将移动应用开发范式从产品经理驱动转向终端用户驱动的可行性。

## Abstract
Software malleability allows applications to be easily changed, configured, and adapted even after deployment. While prior work has explored configurable systems, adaptive recommender systems, and malleable GUIs, these approaches are often tailored to specific software and lack generalizability. In this work, we envision per-user malleable mobile applications, where end-users can specify requirements that are automatically implemented via LLM-based code generation. However, realizing this vision requires overcoming the key challenge of designing automated test generation that can reliably verify both the presence and correctness of user-specified functionalities. We propose \tool, a user-requirement-driven GUI test generation framework that incrementally navigates the UI, triggers desired functionalities, and constructs LLM-guided oracles to validate correctness. We build a benchmark spanning six popular mobile applications with both correct and faulty user-requested functionalities, demonstrating that \tool effectively validates per-user features and is practical for real-world deployment. Our work highlights the feasibility of shifting mobile app development from a product-manager-driven to an end-user-driven paradigm.

### 1. 核心问题与整体含义（研究动机和背景）
*   **背景**：传统的移动应用开发遵循“需求-实现-发布”的静态周期，用户只能被动接受产品经理定义的功能。随着大语言模型（LLM）代码生成能力的提升，未来可能实现“每用户定制化”（Per-User Malleable Apps），即用户通过自然语言提出需求，AI 自动修改源码并生成专属安装包。
*   **核心问题**：在这种高频、个性化的开发范式下，如何确保 AI 生成的功能既**存在**又**正确**？由于应用是闭源且海量的，人工测试不可行，现有的单元测试或通用 UI 测试工具无法理解用户特定的语义意图，因此需要一种能够根据自然语言需求自动生成测试脚本和断言（Oracle）的框架。

### 2. 方法论：Aladdin 框架
Aladdin 的核心思想是将测试过程分为三个阶段，利用 LLM 的语义理解能力引导 GUI 探索和验证：

*   **阶段 1：功能触发状态导航（Functional Triggering State Navigation）**
    *   **目标**：从首页导航到能够触发新功能的 UI 页面。
    *   **技术细节**：采用启发式评分机制。LLM 对当前页面的 UI 元素进行语义相关性评分（$\gamma$），计算路径的几何平均分（$\Gamma$）来决定优先级。这避免了盲目的 BFS/DFS 搜索，优先探索最可能包含新功能入口的路径。
*   **阶段 2：功能存在性检查与执行（Functional Presence Check & Execution）**
    *   **目标**：验证功能入口是否存在，并生成交互脚本触发功能。
    *   **技术细节**：LLM 根据 UI 树生成 `uiautomator2` 脚本（如点击、输入），并生成中间断言确保 UI 元素符合预期。
*   **阶段 3：功能正确性断言生成（Functional Correctness Oracle Generation）**
    *   **目标**：验证执行后的结果是否真正满足了用户需求。
    *   **技术细节**：对比执行前（Pre-state）和执行后（Post-state）的 UI 差异，生成多个子断言（Sub-oracles）来全面捕捉功能行为的正确性。
*   **关键辅助机制：选择器精炼（Selector Refinement）**
    *   针对 LLM 容易产生的 UI 选择器幻觉（如错误的 ID 或文本），通过层级搜索和模糊匹配自动修复失效的脚本定位符。
