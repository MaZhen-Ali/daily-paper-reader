## 摘要
使用 AI 编程智能体的规范驱动开发 (SDD) 提供了一种结构化工作流，但智能体在大型且不断演进的代码库中往往处于“上下文盲”状态，导致产生幻觉 API 和违反架构规范。我们提出了 Spec Kit Agents，这是一个多智能体 SDD 流水线（包含产品经理和开发人员角色），它增加了阶段级的上下文增强钩子。只读探测钩子将每个阶段（规范、计划、任务、实现）锚定在代码库证据中，而验证钩子则根据环境检查中间产物。我们在五个代码库中对涵盖 32 个功能的 128 次运行进行了评估。结果表明，上下文增强钩子在 1-5 分的综合 LLM 评委评分中将质量提升了 0.15 分（占总分的 3.0%；Wilcoxon 符号秩检验，p < 0.05），同时保持了 99.7-100% 的代码库级测试兼容性。我们进一步在 SWE-bench Lite 上评估了该框架，增强钩子将基准性能提升了 1.7%，达到了 58.2% 的 Pass@1。

## Abstract
Spec-driven development (SDD) with AI coding agents provides a structured workflow, but agents often remain "context blind" in large, evolving repositories, leading to hallucinated APIs and architectural violations. We present Spec Kit Agents, a multi-agent SDD pipeline (with PM and developer roles) that adds phase-level, context-grounding hooks. Read-only probing hooks ground each stage (Specify, Plan, Tasks, Implement) in repository evidence, while validation hooks check intermediate artifacts against the environment. We evaluate 128 runs covering 32 features across five repositories. Context-grounding hooks improve judged quality by +0.15 on a 1-5 composite LLM-as-judge score (+3.0 percent of the full score; Wilcoxon signed-rank, p < 0.05) while maintaining 99.7-100 percent repository-level test compatibility. We further evaluate the framework on SWE-bench Lite, where augmentation hooks improve baseline by 1.7 percent, achieving 58.2 percent Pass@1.

论文针对 AI 编程智能体在处理大型、动态演进的代码库时存在的**“上下文盲区”（Context Blindness）**问题展开研究。尽管现有的规范驱动开发（SDD）流程（如 GitHub 的 Spec Kit）通过将任务分解为“规范-计划-任务-实现”等阶段来提高结构化程度，但智能体仍经常生成不存在的 API、错误的路径或违反项目架构规范的代码。这种“幻觉”会导致错误在工作流中不断累积，最终导致任务失败。

### 2. 方法论：核心思想与关键技术
论文提出了 **Spec Kit Agents**，一个多智能体 SDD 流水线，其核心思想是在工作流的每个阶段引入**上下文增强层（Context-Grounding Layer）**：
*   **多角色协作**：包含产品经理（PM）和开发人员（Developer）角色，由状态机编排器管理。
*   **发现钩子（Discovery Hooks）**：在每个阶段开始前运行的只读探测器。它通过 `grep`、`git history` 等工具收集代码库证据（如现有 API、命名规范、依赖关系），确保后续生成基于真实代码现状。
*   **验证钩子（Validation Hooks）**：在每个阶段结束后运行。它检查中间产物（如 `PLAN.md`）中的文件路径是否存在、库是否可用，并在实现阶段后运行测试（如 `pytest`）和静态检查（Linter）。
*   **阶段性约束**：将增强逻辑从主提示词（Prompt）中剥离，作为独立的外部操作，实现了更透明的审计和最小权限控制。
